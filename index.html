System Architecture

The full stack, end to end

Five integrated layers. Every AI interaction captured, streamed, refined, stored, and surfaced back as sovereign enterprise context.

Core Principles

Built on six non-negotiables

Every architectural decision flows from these commitments.

P-01

Sovereignty first

Enterprise knowledge never leaves your control. The ECE layer is vendor-agnostic by design — swap any LLM without losing a single learned pattern.

P-02

Zero-friction capture

The Agent Gateway is a transparent proxy. No behaviour change required from users or developers. Every AI interaction is captured by default.

P-03

Stream-first processing

Kafka ensures knowledge is available within seconds of being generated. No batch jobs, no stale context, no overnight pipelines.

P-04

Knowledge that improves

Context feedback loops mean the repo learns which knowledge chunks produce good outcomes. Retrieval ranking improves with every interaction.

P-05

Governed by design

RBAC, audit logging, data residency enforcement, and schema registries are structural — not bolted on. Compliance is architectural, not operational.

P-06

Tiered agent memory

Memory is structured across three tiers — in-context (session), working (project-scoped), and long-term (enterprise repo) — so agents retrieve at the right granularity and freshness for every query. Decay and promotion are automatic.

Layer Deep Dive

Five layers, one learning loop

Each layer has a single job. Together they compound.

Agent Gateway
The enforcement boundary
proxy · router · enricher

What it does

A transparent reverse proxy that sits between every enterprise client and every LLM endpoint. No client configuration required — traffic is routed through the gateway at the network level. Every outbound request is intercepted, scored for intent and risk, enriched with context from the MCP server, then forwarded. Every response is intercepted, logged, and emitted to Kafka.

The prompt router

Not every query needs enrichment. An intent classifier inside the gateway routes trivial queries (code formatting, generic explanations) directly to the LLM with zero latency overhead. Only domain-specific or sensitive queries trigger deep context retrieval — keeping p99 latency tight.

AgentTrust integration

The gateway is the natural host for egress control, kill switches, and risk scoring from AgentTrust. Every request is scored before enrichment. High-risk requests are blocked, rerouted, or flagged — all at a single enforcement boundary.

Kafka Stream Layer
The real-time refinery
stream · process · publish

Three topics, one pipeline

raw-interactions receives every gateway event at ingestion speed. Stream consumers fan out independently: a Classifier tags domain, intent, and sensitivity; a Synthesiser extracts decisions, patterns, and lessons; an Evaluator scores confidence and applies a quality gate. Output flows to enriched-knowledge.

The feedback topic

context-feedback is the self-improvement mechanism. When the gateway captures which context chunks were used and whether the interaction outcome was positive, those signals flow back here. Retrieval ranking for high-performing chunks is boosted automatically.

Schema registry

A Confluent-compatible schema registry enforces event contracts between gateway producers and stream consumers. Schema drift on raw-interactions is a breaking change — the registry prevents it from silently corrupting downstream consumers.

Knowledge Repo
The institutional memory
tier · promote · expire

Three-tier memory architecture

Agent memory is structured across three distinct tiers, each with its own storage backend, TTL policy, and retrieval latency profile. Tier 1 — In-context memory is ephemeral session state held in the active prompt window: the current conversation, tool call history, intermediate reasoning. Lives only for the duration of the agent session. Tier 2 — Working memory is project-scoped, persisted across sessions within a defined scope (a sprint, a customer engagement, a running incident). Stored in a fast key-value or short-horizon vector index with a configurable TTL of hours to weeks. Tier 3 — Long-term enterprise memory is the durable institutional knowledge repo: domain-classified embeddings, concept graph, versioned decision log, and private evals — partitioned by domain, access-controlled, and retained indefinitely with managed decay.

Promotion and decay

A memory lifecycle agent continuously monitors Tier 2 working memory. Patterns that recur across multiple sessions, or working memory explicitly flagged by a human, are promoted to Tier 3 and classified into the enterprise knowledge graph. Tier 3 chunks that age without retrieval have their confidence scores progressively downgraded — high-importance chunks surface in a human review queue before expiry rather than being silently dropped. This keeps the long-term repo free of stale context without manual curation.

Multi-modal storage

Tier 2 uses a fast vector index (sub-10ms) optimised for recency. Tier 3 uses a full vector store for semantic retrieval, a concept graph for relationship mapping, a versioned decision log for auditability, and a private evals store for business-outcome benchmarks — all domain-partitioned and access-controlled via Vault and OPA.

MCP Server
The retrieval interface
retrieve · govern · audit

Two planes, one interface

The MCP server is split into a retrieval plane (fast vector similarity, sub-50ms SLA, called in the gateway hot path) and a governance plane (RBAC checks, audit logging, data residency enforcement — asynchronous). This keeps enrichment latency tight while maintaining full compliance posture.

Vault + Policy Engine

Secret management and credential rotation for all LLM API keys and service tokens are handled by HashiCorp Vault, with dynamic secret leases so no long-lived credentials exist in config. Access policy decisions — which user, role, or agent can retrieve which knowledge domain — are enforced by Open Policy Agent (OPA), with policies expressed as code and version-controlled alongside the architecture. IBM IAM provides the identity layer for enterprise SSO integration.

Integrated into the gateway

The MCP server is not a standalone tool — it is called by the gateway before every outbound LLM request that passes the intent router's enrichment threshold. Context is injected into the prompt at the gateway level. The LLM receives an already-enriched request and has no awareness of the enterprise context layer.

IBM Reference Implementation

ECE on the IBM stack

REFERENCE IMPLEMENTATION

The open ECE architecture maps cleanly onto IBM's enterprise platform portfolio. Each abstract layer resolves to a production-grade IBM product — giving enterprises a fully governed, supported, and Cloud Pak-native deployment path while remaining architecturally interchangeable with open alternatives.

This is not a proprietary fork. The interfaces remain open. The IBM stack is one opinionated, enterprise-validated instantiation of the reference architecture.

The five layers of ECE resolve to: Confluent Platform for stream processing, watsonx.data OpenSearch for vector retrieval, watsonx.data openRAG for knowledge graph, watsonx Orchestrate as the agent control plane, and a governance layer combining HashiCorp Vault, Open Policy Agent, and IBM IAM for zero-trust RBAC across every boundary.

All components are deployable on Red Hat OpenShift via Cloud Pak for Data, with full data residency control for APAC regulatory environments.

Agent Gateway
Control plane
open: nginx / envoy proxy
watsonx Orchestrate
Acts as the agent control plane — orchestrating tool calls, managing agent sessions, enforcing skill boundaries, and routing enriched prompts to the correct LLM endpoint. The Orchestrate agent runtime replaces a custom proxy for AI-native workloads, with built-in observability and tool-use tracing.
HashiCorp Vault
Secret injection at the gateway boundary. All LLM API keys, service tokens, and database credentials are dynamically issued per-request via Vault's secrets engine. No long-lived credentials in config or environment variables. Vault agent sidecar handles automatic renewal and rotation.
OPA Policy Engine
Request-time policy enforcement. Every inbound request is evaluated against OPA policies before enrichment — checking user identity, data classification, model access rights, and jurisdiction constraints. Policies are expressed as Rego, version-controlled, and hot-reloaded without gateway restarts.
Stream Layer
Real-time pipeline
open: apache kafka
Confluent Platform
Enterprise Kafka with Schema Registry, ksqlDB, and Kafka Connect built in. Confluent's Schema Registry enforces Avro/Protobuf contracts on the raw-interactions and enriched-knowledge topics — preventing schema drift across classifier, synthesiser, and evaluator consumers. ksqlDB enables stream-processing queries without a separate Flink or Spark cluster. Fully deployable on OpenShift via Confluent for Kubernetes operator.
Knowledge Repo
Institutional memory
open: postgres + weaviate
watsonx.data OpenSearch
Vector store and semantic retrieval layer. OpenSearch's k-NN plugin provides ANN-based vector similarity search across embedded knowledge chunks — the primary retrieval path for the MCP server's context_lookup tool. Integrated natively into the watsonx.data lakehouse, with column-level access control via IBM IAM and full audit logging to the data governance layer.
watsonx.data openRAG
Knowledge graph and RAG pipeline orchestration. openRAG manages entity extraction, relationship mapping, and graph-based retrieval across the enterprise concept graph — surfacing multi-hop reasoning that flat vector search cannot reach. Handles the synthesiser consumer's output: linking decisions to their evidence chains, mapping domain concepts to related patterns, and enabling the decay agent to propagate confidence changes across connected nodes.
IBM IAM
Identity layer for domain-partitioned RBAC. IBM IAM enforces access boundaries across knowledge domains — a finance query cannot retrieve engineering knowledge, an external agent cannot access internal decision logs. IAM integrates with OPA: IAM provides the identity assertion, OPA applies the policy. All access decisions are logged to the watsonx.data audit store.
MCP Server
Retrieval interface
open: mcp protocol
watsonx Orchestrate
MCP tool host and skill registry. Orchestrate exposes the ECE knowledge repository as native MCP tools — context_lookup, decision_log, knowledge_search — which any connected LLM or agent can invoke. The retrieval plane (OpenSearch k-NN) and governance plane (OPA + IAM) are wired as separate Orchestrate skills with distinct SLA profiles, keeping the hot retrieval path independent of asynchronous compliance checks.
HashiCorp Vault
Credential management for MCP tool endpoints. Service-to-service credentials between the MCP server, OpenSearch, openRAG, and external LLM APIs are managed via Vault's PKI and database secrets engines. Mutual TLS between all components. Certificate rotation is automatic and zero-downtime.
OPA Policy Engine
Retrieval-time policy evaluation. Before any context chunk is returned by the MCP server, OPA evaluates the requesting agent's identity, the chunk's classification, and the applicable data residency policy. Chunks flagged as jurisdiction-restricted are filtered before they reach the LLM prompt. Policy violations are emitted to Confluent for real-time alerting.
